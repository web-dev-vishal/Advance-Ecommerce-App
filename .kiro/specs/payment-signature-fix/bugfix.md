# Bugfix Requirements Document

## Introduction

The Razorpay payment signature verification in ShopNest's backend is broken, causing all payment verifications to fail. There are three compounding issues: (1) `RAZORPAY_KEY_ID` and `RAZORPAY_KEY_SECRET` are empty in `.env`, meaning the HMAC is computed with an empty secret and the Razorpay SDK cannot authenticate; (2) the `/api/payment/verify` route receives a raw `Buffer` body due to `express.raw()` being applied before `express.json()`, but the controller's Buffer-to-JSON parsing path is fragile and can silently fail; (3) the signature comparison uses `Buffer.from(razorpay_signature, 'hex')` — but Razorpay returns the signature as a hex string, so decoding it from hex before comparison is correct only if the expected signature is also decoded from hex, which it is — however if the two hex strings differ in length (e.g. due to empty secret producing a different digest length), `timingSafeEqual` throws instead of returning false, crashing the request.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN `RAZORPAY_KEY_SECRET` is an empty string THEN the system SHALL compute the HMAC with an empty key, producing a signature that never matches the one generated by Razorpay's servers

1.2 WHEN a POST request is sent to `/api/payment/verify` with `Content-Type: application/json` THEN the system receives a raw `Buffer` (due to `express.raw()` intercepting the route before `express.json()`) and the controller must manually parse it, introducing a failure point if the buffer is malformed or empty

1.3 WHEN `razorpay_signature` is a valid 64-character hex string but `RAZORPAY_KEY_SECRET` is empty THEN `expectedSign` is a 64-character hex string derived from an empty-key HMAC, both `Buffer.from(..., 'hex')` calls produce 32-byte buffers, `timingSafeEqual` runs but returns `false` — however if either hex string has an odd length or non-hex characters, `Buffer.from` silently produces a buffer of wrong length and `timingSafeEqual` throws a `RangeError`, causing a 500 response instead of a 400

1.4 WHEN `RAZORPAY_KEY_ID` is empty THEN the system creates a Razorpay SDK instance with no key, causing `createOrder` to fail with an authentication error from Razorpay's API

### Expected Behavior (Correct)

2.1 WHEN `RAZORPAY_KEY_SECRET` is set to a valid non-empty secret THEN the system SHALL compute the HMAC-SHA256 signature correctly and it SHALL match the signature provided by Razorpay for a legitimate payment

2.2 WHEN a POST request is sent to `/api/payment/verify` THEN the system SHALL parse the request body as JSON regardless of whether it arrives as a raw `Buffer` or a pre-parsed object, without throwing on malformed input

2.3 WHEN `razorpay_signature` and the computed `expectedSign` are both valid hex strings of equal length THEN the system SHALL perform a constant-time comparison and return `200 Payment verified successfully` for a matching signature

2.4 WHEN `razorpay_signature` and `expectedSign` differ in byte-length (e.g. due to invalid hex input) THEN the system SHALL return `400 Invalid payment signature` without throwing a `RangeError` or returning a 500 error

2.5 WHEN `RAZORPAY_KEY_ID` is set to a valid key THEN the system SHALL successfully create a Razorpay order and return the order object to the client

### Unchanged Behavior (Regression Prevention)

3.1 WHEN a request to `/api/payment/verify` is missing any of `razorpay_order_id`, `razorpay_payment_id`, or `razorpay_signature` THEN the system SHALL CONTINUE TO return `400` with a descriptive validation message

3.2 WHEN a valid amount is provided to `POST /api/payment/order` THEN the system SHALL CONTINUE TO return a deduplicated Razorpay order (using the Redis cache) for the same user and amount within 10 minutes

3.3 WHEN an invalid amount (zero or negative) is provided to `POST /api/payment/order` THEN the system SHALL CONTINUE TO return `400 Valid amount is required`

3.4 WHEN a payment is successfully verified THEN the system SHALL CONTINUE TO publish a `payment.verified` event to RabbitMQ with `razorpay_order_id`, `razorpay_payment_id`, and `timestamp`

3.5 WHEN a request reaches `/api/payment/verify` or `/api/payment/order` without a valid JWT THEN the system SHALL CONTINUE TO return `401 Unauthorized` via the `protect` middleware

## Bug Condition Pseudocode

### Bug Condition Function

```pascal
FUNCTION isBugCondition(X)
  INPUT: X of type PaymentVerifyRequest
  OUTPUT: boolean

  // Bug is triggered when any of the following hold:
  RETURN (RAZORPAY_KEY_SECRET = "") 
      OR (X.body is Buffer AND JSON.parse fails)
      OR (Buffer.from(X.razorpay_signature, 'hex').length ≠ Buffer.from(expectedSign, 'hex').length)
END FUNCTION
```

### Property: Fix Checking

```pascal
// For all requests that previously triggered the bug
FOR ALL X WHERE isBugCondition(X) DO
  result ← verifyPayment'(X)
  ASSERT result.status ∈ {200, 400}   // never 500
  ASSERT result does NOT throw RangeError
END FOR
```

### Property: Preservation Checking

```pascal
// For all well-formed requests with valid credentials
FOR ALL X WHERE NOT isBugCondition(X) DO
  ASSERT verifyPayment(X) = verifyPayment'(X)
END FOR
```
